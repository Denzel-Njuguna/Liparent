liparent_v1 Project
File Structure & Explanation

Overview
This is a Phoenix (Elixir) web application. The project is split into two layers: the core business logic layer (liparent_v1/) and the web layer (liparent_v1_web/). Below is an explanation of every file in the lib/ directory.

Core Application Files
liparent_v1.ex
The core application module definition. Similar to the web file above but for the non-web (business logic) layer. Defines shared macros for schemas, contexts, and other core modules. It also exposes child_spec/1 for OTP supervision.

liparent_v1/application.ex
The OTP Application entry point. This is what starts when your app boots. It defines the supervision tree - starting the Repo (database), Telemetry, the PubSub system, and the Endpoint (web server). If you add background workers, caches, or other long-running processes, you register them here.

liparent_v1/mailer.ex
The mailer module, generated by Swoosh. It defines the mailer adapter your app uses to send emails (e.g., SMTP, SendGrid, Mailgun). You call functions from this module - like deliver/1 - to actually send emails built with your Email modules.

liparent_v1/repo.ex
The Ecto Repo. This is your database interface. All database queries, inserts, updates, and deletes go through Repo. It wraps Ecto.Repo and connects to your configured database (PostgreSQL by default). It is started as a child process in application.ex.

Web Layer Files
liparent_v1_web.ex
The web module definition file. It defines shared macros (use liparent_v1_web, :controller), (use liparent_v1_web, :router), etc. that inject common imports and aliases into controllers, views, routers, and channels across the entire web layer. Think of it as the base configuration blueprint for every web-related module.

liparent_v1_web/controllers/error_json.ex
A JSON error view controller. It handles rendering structured JSON error responses for HTTP errors like 404 (Not Found) and 500 (Internal Server Error). When your API returns an error status code, this file controls what the JSON body looks like.

liparent_v1_web/endpoint.ex
The Phoenix Endpoint - the entry point for all HTTP requests. It is the first Plug in the pipeline. It sets up things like static file serving, session handling, code reloading in dev, CSRF protection, and the router. Every HTTP request passes through here before reaching your router.

liparent_v1_web/gettext.ex
The Gettext module for internationalisation (i18n). It lets you mark strings for translation using gettext/1 and ngettext/3 macros. Translation files (.po files) live in priv/gettext/. If your app is English-only right now, this file sits quietly unused.

liparent_v1_web/router.ex
The router. It maps incoming HTTP requests (method + path) to controller actions. It defines pipelines (groups of plugs like :browser or :api) and scopes (URL prefixes like /api). This is where you add new routes as your app grows.

liparent_v1_web/telemetry.ex
The Telemetry supervisor. It sets up metrics and instrumentation for your app - tracking things like request durations, database query times, and VM memory usage. It wires Phoenix and Ecto telemetry events to reporters (e.g., LiveDashboard). Useful for monitoring and observability.

How a Request Flows Through the App
When a user hits your API, here is the order files are involved:
1.  endpoint.ex - receives the raw HTTP request first
2.  router.ex - matches the path and method to a controller action
3.  Your controller - handles the request, calls context/business logic
4.  repo.ex - if data is needed, the controller/context queries the database through Repo
5.  error_json.ex - if something goes wrong, this formats the JSON error response
